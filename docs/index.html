The idea

Durable execution, minus the machinery.

File is the task. Path is the identity. Same path resumes the same task; a different path is a different task. A run is two primitives: step and nap. A done step commits its result and never runs again — later attempts return the cached value. Crash mid-step? The next invocation re-runs only what didn't finish.

No daemon. No broker. No scheduler. Bring your own cron.

And it's one file. journal_mode=DELETE, so no -wal/-shm siblings — the file is the whole artifact. cp it, mv it, rm it, tarball it, crack it open with sqlite3. Backups: copy the file. Done.

Inspired by Absurd — same "durable workflows are absurdly simple" thesis, Postgres swapped for SQLite, so a workflow is one file you can hold.

Quickstart

Zero to resumable.

Wrap each expensive step. Run it. Kill it mid-way. Run again — finished steps replay from one SQLite file instead of re-executing. No re-charge, no re-send. The file is the task.

eval "$(mundane init task.db)" takes the lock, bootstraps the file, defines step and nap. 
#!/bin/sh
eval "$(mundane init task.db)"
step fetch  -- curl -fsS https://api/x
nap  cool   5m
step notify -- ./send-email.sh
mundane.Run holds the file; mundane.Step is generic, so the cached value comes back typed. 
import "github.com/paulbellamy/mundane/go/mundane"

err := mundane.Run("task.db", func(ctx *mundane.Ctx) error {
    user, err := mundane.Step(ctx, "fetch", func() (User, error) {
        return fetchUser(id)
    })
    if err != nil { return err }
    return ctx.Sleep("cool", "5m")
})
Plain functions, no decorators. mundane.run drives the body. Async: mundane.arun with ctx.astep / ctx.asleep. 
import mundane

def workflow(ctx):
    user = ctx.step("fetch", lambda: requests.get(url).json())
    ctx.sleep("cool-off", "5m")
    ctx.step("notify", lambda: send_email(user["email"], "hi"))

mundane.run("task.db", workflow)
await run(...) with await ctx.step(...) and await ctx.sleep(...). Returns the body's value. 
import { run } from "mundane-sdk";

await run("task.db", async (ctx) => {
  const user = await ctx.step("fetch", async () => (await fetch(url)).json());
  await ctx.sleep("cool-off", "5m");
  await ctx.step("notify", async () => { await sendEmail(user.email, "hi"); return null; });
});

Second run reopens the same file and reads the rows. Steps marked done return their cached result instantly — fetch doesn't re-fetch, notify doesn't re-send. Execution skips straight to where the crash hit. The API call you already paid for stays paid once.

Two primitives. That's the whole library.

step(name, fn)

Runs fn once on a cache miss, caches the JSON result, returns it on every later attempt. Never runs twice for a done step. For work you want at most once: API call, LLM call, email, write.

nap / sleep(name, duration)

Durable sleep. First call records wake_at = now + duration. A resume after a crash sleeps only the remaining time, not the whole duration again. Takes "30s", "5m", "2h", or ms. Shell calls it nap; SDKs call it ctx.sleep.

Rules that bite

The fine print, up front.

Break one and durability breaks with you.

  • Steps must be idempotent.

    Crash mid-step leaves the row pending; next invocation re-runs it. The contract you sign.

  • Step names are strict.

    Match ^[A-Za-z0-9][A-Za-z0-9._-]*$. No spaces, slashes, colons, unicode. Duplicate in one body: DuplicateStepError, exit 2.

  • Results must JSON round-trip.

    Date, undefined, BigInt, Map/Set, functions, circular refs throw on first write — at the call site, not silently corrupted.

  • Retry == re-invoke.

    A step's nonzero exit or thrown error marks it failed and propagates. No built-in retry, backoff, or timeout.

  • One writer per file.

    flock. Second concurrent invocation fails fast: exit 75 / LockedError. No queue. No wait.

When NOT to use it

Small on purpose.

We are proud of the list. Reach elsewhere when you have:

  • —A one-shot command, no expensive or non-idempotent steps. The file is pure overhead.
  • —Request/response or sub-second paths. Every step is a synchronous SQLite write plus flock.
  • —Concurrency on one task. One writer, period. No fan-out, no parallel, no cross-task signals.
  • —A real queue, scheduler, broker, or worker pool. No scheduler here; long naps block the process.
  • —Built-in retry, backoff, or timeout. None exist. Retry == re-invoke.
  • —Steps you can't make idempotent, or results that aren't JSON.
  • —Schema migrations or multiple versions. Files pin to v1; mismatch is a hard error.
